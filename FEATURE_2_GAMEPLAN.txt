======================================================
AUTOMATION 1 - PROJECT STATUS EMAIL
======================================================

WHAT IT DOES
------------------------------------------------------
User selects a single Q360 dispatch record. Clicks Generate. Claude reads
the dispatch data and writes a complete professional client-facing status
update email - subject line and full body - streamed live to the screen
character by character.


Q360 DATA USED
------------------------------------------------------
  Customer name and contact email
  Project or dispatch title and number
  Current status code (OPEN, IN PROGRESS, CLOSED, etc.)
  Technician assigned
  Problem description
  Solution notes (if any)
  Estimated fix time
  Open date and close date


AI MODEL
------------------------------------------------------
Model: claude-sonnet-4-6

Why Sonnet over Haiku:
  Sonnet produces emails that read like a skilled account manager wrote
  them. Haiku produces functional but flat output. For a live sponsor demo
  where the email is the centerpiece, quality wins. The cost difference
  (~$0.01 vs $0.003 per call) is irrelevant at demo scale.

Streaming: YES
  Text streams to the browser token by token. This makes it look like the
  AI is typing the email live. Visually impressive. Much better than a
  spinner followed by a wall of text appearing all at once.

max_tokens: 600
  A well-structured professional email runs 200-400 tokens. 600 gives
  headroom without risking runaway generation.


PROMPT STRATEGY
------------------------------------------------------
Split between system prompt and user prompt:

  System prompt  ->  Persona, output format rules, tone guide
                     This is stable across every call.

  User prompt    ->  The formatted Q360 dispatch data + tone selection
                     This changes every call.

NEVER inject raw Q360 JSON into the prompt. The field names like
disp_status_cd and sched_date_end are machine-readable noise. Pre-format
the data into labeled plain text before injecting:

  Customer: Acme Corporation (john@acme.com)
  Project: HVAC Maintenance - Dispatch D-2041
  Status: IN PROGRESS
  Technician Assigned: Maria Chen
  Problem: Unit 3 compressor cycling irregularly
  Solution Notes: Replaced capacitor on Unit 3
  Estimated Completion: March 30, 2026
  Opened: March 25, 2026
  Closed: [Not provided]

Null or empty fields always become the literal text [Not provided].
This tells Claude to omit or handle gracefully, never make something up.

Output format Claude must follow:
  Line 1:     SUBJECT: [subject line here]
  Line 2:     (blank line)
  Line 3+:    Full email body starting with greeting

The frontend parses the SUBJECT: line out separately and displays it
above the email body textarea.

Tone parameter options:
  Professional  ->  Formal business language, no contractions
  Friendly      ->  Warm, light contractions, empathetic
  Concise       ->  Short sentences, bullets, under 150 words


API ROUTE - /app/api/feature2/generate/route.ts
------------------------------------------------------
Method: POST
Request body: { recordType, recordId, automationType, tone }

Steps:
  1. Validate recordId and tone are present
  2. Call fetchDispatch(recordId) from q360Client.ts
     If Q360 is unreachable, return HTTP 502 with friendly error message
  3. Call formatDispatchForPrompt(data) to build clean text block
  4. Build user prompt by injecting formatted data and tone
  5. Call anthropic.messages.stream() with claude-sonnet-4-6
  6. Return ReadableStream to the browser
     Headers: Content-Type: text/plain, Transfer-Encoding: chunked

Error handling:
  Q360 down        ->  HTTP 502, "Q360 API unavailable"
  Claude fails     ->  Stream pushes error sentinel text at end
  Bad request      ->  HTTP 400, "recordId and tone are required"
  Unknown error    ->  HTTP 500, logged to console

HOW THE FRONTEND READS THE STREAM
------------------------------------------------------
  1. POST to /api/feature2/generate
  2. Get response.body as a ReadableStream
  3. Read chunks with response.body.getReader()
  4. Decode each chunk with TextDecoder
  5. Once a blank line follows the SUBJECT: line:
       Parse subject line into the subject bar state
       Begin appending remaining text to the body textarea state
  6. Each state update triggers a React re-render
     The textarea appears to fill in live
  7. When reader.done === true:
       Set isStreaming to false
       Make textarea editable
       Show Copy and Regenerate buttons

Fallback: If no SUBJECT: line is detected after the full response,
set a generic subject ("Service Update - Dispatch #ID") and show
the full buffer as the body.


THE THREE DEMO MOMENTS TO NAIL
------------------------------------------------------
1. The dropdown populates with real Q360 records
   Sponsor sees their own customer names and dispatch numbers.
   This sets credibility - it is not a mockup.

2. The subject line appears first, then the body streams in
   The two-beat reveal shows the AI understood the task structure.
   If both appear simultaneously as one blob, the effect is flat.

3. Switching tone and clicking Regenerate gives a noticeably different email
   This proves the system is not cached and the AI is reasoning.
   Sponsors often ask "can it be less formal?" - answer it live.


EDGE CASES
------------------------------------------------------
Null Q360 field:
  formatDispatchForPrompt() converts null/undefined/empty to [Not provided]
  Claude is instructed to omit gracefully, never fabricate

No SUBJECT: line from Claude:
  Fall back to generic subject: "Service Update - Dispatch #ID"
  Log console.warn so developers know the prompt regressed

Q360 down during demo:
  Seed 3-5 static dispatch records as JSON fallbacks in q360Client.ts
  If the real Q360 call fails, return the static seed for that record ID
  Add console.warn("Using demo fallback data")
  The demo never breaks

Claude stream interrupted mid-response:
  Push error sentinel to stream: [ERROR: Generation interrupted. Regenerate.]
  Frontend detects sentinel and shows inline error instead of broken text


======================================================
AUTOMATION 3 - OVERDUE DISPATCH ALERT
======================================================

WHAT IT DOES
------------------------------------------------------
User clicks one button. The app fetches ALL open dispatches from Q360,
computes which ones are past their estimated fix time, sends the batch to
Claude for analysis, and displays a color-coded dashboard of alert cards
showing which calls need immediate attention - ranked by urgency with an
AI-written summary and recommended action per call.


Q360 DATA USED
------------------------------------------------------
  All open dispatch records (bulk fetch, paginated)
  estfixtime field vs. current date (overdue comparison)
  Dispatch creation date (fallback for overdue calculation)
  Customer name and site
  Assigned technician (or unassigned)
  Problem description and priority level
  How many days overdue (computed server-side)


AI MODEL
------------------------------------------------------
Model: claude-sonnet-4-6

Why Sonnet:
  The entire demo value of this feature is Claude doing intelligent
  prioritization - not just sorting by a number, but synthesizing
  customer context, priority level, problem description, and days overdue
  into coherent urgency rankings with specific actionable summaries.

  Haiku would rank correctly but produce flat summaries.
  Sonnet produces output like:
    "Machine-down call with no tech assigned - client waiting 2 weeks.
     Escalate to dispatch manager immediately."

  At 50 records that is roughly 8,000 input tokens + 3,000 output tokens.
  Cost per scan: roughly $0.07. Still irrelevant for a demo.

Streaming: NO (for the JSON response)
  The JSON must be complete before any cards can render.
  Streaming provides no UX benefit for structured data.
  Use a two-phase loading animation instead:
    Phase 1: "Fetching open dispatches from Q360..."
    Phase 2: "AI is analyzing X overdue dispatches..."

max_tokens: 3000
  JSON output for 50 alerts with summaries and recommended actions.


OUTPUT FORMAT
------------------------------------------------------
Claude returns a JSON object (not plain text, not an email):

{
  "summary": {
    "totalScanned": 45,
    "totalOverdue": 8,
    "critical": 2,
    "high": 4,
    "medium": 2
  },
  "alerts": [
    {
      "dispatchno": "T223225520",
      "urgencyTier": "CRITICAL",
      "daysOverdue": 14,
      "customer": "Anixter Inc.",
      "site": "Wauwatosa, WI",
      "problem": "Hardware equipment failure",
      "techAssigned": null,
      "aiSummary": "Machine-down call with no tech assigned - client waiting 2 weeks.",
      "recommendedAction": "Assign technician and contact customer within 1 hour."
    }
  ]
}

Sorted by: CRITICAL first, then HIGH, then MEDIUM.
Within each tier: most overdue first.


URGENCY TIERS
------------------------------------------------------
CRITICAL:   14+ days overdue
            OR no tech assigned AND 7+ days overdue
            OR priority = 1

HIGH:       7-13 days overdue

MEDIUM:     1-6 days overdue


Q360 DATA FETCHING STRATEGY
------------------------------------------------------
Endpoint: POST /api/Record/dispatch?_a=list
Auth: Basic Auth header (server-side only, never exposed to browser)

Request body:
  columns: dispatchno, callno, customerno, statuscode, date, estfixtime,
           priority, problem, problemcode, detail, techassigned,
           customer_company, site_sitename, site_address, site_city,
           site_state, site_zip, tech_fullname, csr
  filters: statuscode = OPEN
  orderBy: date ASC
  offset: 0
  limit: 100

Pagination:
  Check response: payload.outvars.hasmore === "Y"
  If true: increment offset by 100, fetch again
  Loop until hasmore is not Y or records returned < limit
  Collect all pages into one array


COMPUTING OVERDUE SERVER-SIDE
------------------------------------------------------
Today's date is always computed in the API route from new Date().
Never use the client's date.

Step 1: Try estfixtime
  If the field is not null, not ".00", not empty, and parses as a valid
  date: compute (today - estfixtime) in days.

Step 2: Fallback (needed for the sandbox)
  The sandbox shows estfixtime: ".00" on all records. It is not a usable
  datetime. Fallback: compute (today - (dispatch.date + 7 days)).
  Any call open more than 7 days counts as overdue under SLA rules.

A dispatch is overdue if computed days > 0.


API ROUTE - /app/api/feature2/overdue/route.ts
------------------------------------------------------
Method: POST (no body needed)

Steps:
  1. Set today = new Date()
  2. fetchAllOpenDispatches() - paginated Q360 fetch
     If Q360 fails, return HTTP 500 with error message
  3. If 0 dispatches returned: return empty state payload
  4. For each dispatch: computeDaysOverdue(dispatch, today)
  5. Filter to overdue only (daysOverdue > 0)
  6. If 0 overdue: return "all within SLA" payload
  7. Sort by daysOverdue descending
  8. Cap at 50 dispatches (prevents prompt from getting too large)
  9. Map to OverdueDispatch objects
  10. Build prompt with formatDispatchBlock() for each record
  11. Call anthropic.messages.create() - non-streaming - claude-sonnet-4-6
  12. Strip any accidental markdown fences from response
  13. JSON.parse() the response
      If parse fails: return { data: null, rawText: claudeResponse }
  14. Return { success: true, data: parsedJSON }

EDGE CASES
------------------------------------------------------
estfixtime is null or ".00":
  This is the default in the sandbox. Fallback to date + 7-day SLA.
  Handled automatically in computeDaysOverdue().

Zero overdue dispatches:
  Return "all within SLA" payload from API route.
  UI shows green checkmark and reassuring message.
  This is actually a good demo moment if it happens.

50+ overdue dispatches:
  Cap at 50 in the API route (already implemented).
  Show truncation notice in the UI.
  In production, prioritize top 20 + all CRITICAL tier instead.

Q360 returns no dispatches:
  Check status code filter (sandbox may use a different status string).
  Return empty state payload with message.

Claude returns invalid JSON:
  Strip markdown fences and retry parse.
  If still fails, return rawText to frontend.
  UI falls back to showing raw text in a monospace block.

Q360 down during demo:
  Same approach as Automation 1 - seed static overdue dispatch data.
  Fallback activates silently if real fetch fails.
